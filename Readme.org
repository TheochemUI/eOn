A modern, elegant approach to working with [[https://eondocs.org][eOn]].

** Build
*** Pixi

We provide a [[https://prefix.dev][pixi]] environment for reproducible builds:

#+begin_src bash
pixi install
#+end_src

Then build and install:

#+begin_src bash
pixi run bash -c "meson setup bbdir \
  --prefix=$CONDA_PREFIX --libdir=lib \
  --buildtype release --wipe 2>/dev/null \
  || meson setup bbdir --prefix=$CONDA_PREFIX \
  --libdir=lib --buildtype release"
pixi run meson install -C bbdir
#+end_src

The server is accessed through ~python -m eon.server~. The ~eonclient~ binary
is automatically made available in the activated environment.

** Running Tests
*** Meson

This is the preferred method.

#+begin_src bash
cd client
# in pixi, add --prefix=$CONDA_PREFIX --libdir=lib
meson setup bbdir --buildtype=debug -Dbuild_tests=true
meson test -C bbdir
#+end_src

*** Python (ASV Benchmarks)

Performance benchmarks use [[https://asv.readthedocs.io][ASV]] with results
posted on PRs via [[https://github.com/HaoZeke/asv-perch][asv-perch]]:

#+begin_src bash
# Validate benchmarks
pixi run bash -c "pip install asv && asv check"

# Quick local run
pixi run bash -c "asv machine --yes && \
  asv run -E 'existing:$(which python)' --quick"
#+end_src

* Contributing

Run ~pre-commit~ before submitting pull requests:

#+begin_src bash
uvx prek run --all-files
#+end_src

For best results, install the git hook:

#+begin_src bash
uvx prek install
#+end_src

* Testing

We have both ~Catch2~ unit tests, and also ~ApprovalTests~. For adding new
approval tests, first build them with ~meson compile~, run them to approve, and
then re-run to make sure things pass with ~meson test~.
